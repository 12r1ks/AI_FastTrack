# Parking Chatbot — Design Spec

**Date:** 2026-05-22  
**Status:** Approved

---

## Overview

A parking space reservation chatbot with a two-page web UI. Users chat with an AI assistant to get parking info and make reservations. Reservation requests are escalated to a human admin for approval. Confirmed reservations are persisted via an MCP server.

Built stage by stage. All 4 stages share the same LangGraph graph, expanded at each stage.

## LangGraph Graph

### State schema

```python
class State(TypedDict):
    messages: Annotated[list[BaseMessage], add_messages]
    intent: str | None            # "info" | "reservation"
    reservation: dict | None      # collected fields so far
    approval_status: str | None   # "pending" | "approved" | "rejected"
```

### Stage 1 graph

```
START → [router] → info query    → [rag_node]       → END
                 → reservation   → [collect_details] → END
```

- **router**: classifies last user message into `intent`
- **rag_node**: retrieves from FAISS (static) + SQLite (dynamic), generates response
- **collect_details**: checks which fields are missing (`name`, `surname`, `car_number`, `datetime_from`, `datetime_to`), asks for the next one; when complete, asks user to confirm before escalating

### Future stages (stubs, not built in stage 1)

- Stage 2: `[await_admin]` node with `interrupt()` — pauses graph until admin decides
- Stage 3: `[write_reservation]` node — calls MCP server after approval
- Stage 4: full graph wired end-to-end

### Checkpointer

`SqliteSaver` on `data/checkpoints.db`. `thread_id` = frontend session UUID. Required for `interrupt()` in stage 2.

## Data Design

### Static → FAISS

Seeded once from `app/seed/static_seed.py`. Documents embedded with metadata:

| Category | Content |
|---|---|
| `general` | Parking name, description |
| `location` | Address, directions |
| `hours` | Opening hours |
| `pricing` | Rate tiers |
| `rules` | Booking policies, restrictions |
| `faq` | Common questions |

### Dynamic → SQLite

**`parking_spaces`** — inventory  
`id`, `space_number`, `zone`, `is_available`

**`availability_slots`** — bookable windows  
`id`, `space_id` (FK), `datetime_from` (ISO), `datetime_to` (ISO), `status` (`free|reserved|pending`)

**`reservations`** — confirmed bookings (written by MCP server in stage 3)  
`id`, `name`, `surname`, `car_number`, `datetime_from`, `datetime_to`, `approved_at`

Seeded from `app/seed/dynamic_seed.py`.

## FastAPI API

| Method | Path | Purpose |
|---|---|---|
| `GET` | `/` | Serve `chat.html` |
| `GET` | `/admin` | Serve `admin.html` |
| `POST` | `/chat` | `{session_id, message}` → `{reply, status}` |
| `GET` | `/admin/pending` | List sessions awaiting approval with reservation details |
| `POST` | `/admin/decision` | `{session_id, decision: "approved"\|"rejected"}` |

`session_id` (UUID, generated by frontend on first load) maps 1:1 to LangGraph `thread_id`. Graph is a module-level singleton compiled at startup. No server-side session storage — all state in LangGraph checkpointer.

`status` in `/chat` response: `"chatting" | "awaiting_approval" | "confirmed" | "rejected"`

## Frontend

### chat.html

Classic chat layout:
- Fixed header: app name
- Scrollable message area: user (right, indigo) / bot (left, dark)
- Sticky input bar: text field + Send button
- Session UUID generated on page load, sent with every request

### admin.html

Card list layout:
- Fixed header with pending count badge
- One card per pending reservation:
  - Name, car number, submitted datetime
  - From / To (datetime: `YYYY-MM-DD HH:MM`)
  - Approve (green) / Reject (red) buttons
- Polls `GET /admin/pending` every 5 seconds

## Reservation Flow (end-to-end, all stages)

```
User types message
  → POST /chat {session_id, message}
  → LangGraph graph.invoke({messages}, config={thread_id})
  → router classifies intent
  → if info: rag_node retrieves + responds
  → if reservation: collect_details asks for missing fields
  → when all fields collected: bot confirms details with user
  → user confirms → graph reaches await_admin node → interrupt()
  → /chat returns {status: "awaiting_approval"}
  → admin sees card on /admin/pending
  → admin clicks Approve/Reject → POST /admin/decision
  → graph.invoke(Command(resume="approved"), config)
  → if approved: write_reservation node → MCP server → reservations.txt
  → next /chat poll returns {status: "confirmed"|"rejected"}
```
